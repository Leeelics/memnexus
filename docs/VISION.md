# MemNexus 产品定位与愿景

> **版本**: 1.0  
> **日期**: 2026-03-25  
> **状态**: 重生规划

---

## 一、一句话定位

**MemNexus Code Memory —— 让 AI 编程助手真正记住你的项目**

> 不是多 Agent 协作编排，不是通用记忆层，而是专门为 AI 编程工具设计的代码记忆基础设施。

---

## 二、问题陈述

### 2.1 用户痛点

当开发者使用多个 AI 编程工具（Claude Code、Kimi CLI、Codex、Cursor 等）时，面临以下问题：

| 痛点 | 场景 | 现有解决方案的不足 |
|------|------|-------------------|
| **Session 孤岛** | 昨天用 Claude 写的功能，今天用 Kimi 优化，需要重新解释一遍 | Vibe Kanban 有 Session，但无记忆共享 |
| **代码上下文丢失** | 问 AI "这个函数在哪里被调用"，它只能猜 | 通用 RAG 不懂代码结构 |
| **项目历史遗忘** | "三个月前为什么选这个架构？" | Git 历史没人看，commit message 散落 |
| **工具切换成本** | 每个 AI 工具都是信息孤岛 | 没有跨工具的记忆层 |

### 2.2 市场空白

```
┌─────────────────────────────────────────────────────────────────┐
│                     AI 编程工具栈分层                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Layer 4: 应用层         Vibe Kanban, Claude Code, Kimi CLI      │
│           (项目管理 + 代码执行)                                   │
│                           ↓ 缺少记忆层                           │
│  Layer 3: 编排层         ── 空白 ──                              │
│           (被 VK 覆盖)                                            │
│                           ↓                                      │
│  Layer 2: 记忆层         Mem0, Supermemory, OpenViking           │
│           (通用记忆，不针对编程)                                   │
│                           ↓ 编程专用记忆空白                      │
│  Layer 1: 存储层         LanceDB, ChromaDB, pgvector             │
│                                                                  │
│  【空白】编程专用记忆层：理解代码语义 + Git 历史 + 跨工具共享        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 三、产品定位

### 3.1 我们是什么

**MemNexus Code Memory** 是专门为 AI 编程场景设计的记忆基础设施：

- **Code-Aware**: 理解代码结构（AST），不只是文本
- **Git-Native**: 与 Git 历史深度集成，代码演进可追溯
- **Tool-Agnostic**: 服务所有 AI 编程工具，不绑定单一平台
- **Developer-First**: 为开发者设计，不是为 AI 研究者设计

### 3.2 我们不是什么

| 不是 | 原因 | 竞品 |
|------|------|------|
| 不是通用 Agent 记忆 | 太宽泛，Mem0/Supermemory 做得更好 | Mem0, Supermemory |
| 不是项目管理工具 | Vibe Kanban 已经做得很好 | Vibe Kanban |
| 不是代码编辑器 | Cursor/VSCode 是编辑器 | Cursor, VSCode |
| 不是模型训练工具 | 不涉及模型微调 | - |

### 3.3 差异化价值

```
┌─────────────────────────────────────────────────────────────────┐
│                     差异化对比                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  vs Mem0/Supermemory (通用记忆)                                  │
│  ─────────────────────────────────────────────                   │
│  Mem0: "用户喜欢什么颜色" → 记忆                                 │
│  MemNexus: "这个函数被谁调用、什么时候修改、为什么修改" → 记忆    │
│                                                                  │
│  vs OpenViking (通用上下文数据库)                                 │
│  ─────────────────────────────────────────────                   │
│  OpenViking: 文件系统范式，L0/L1/L2 分层                         │
│  MemNexus: 代码语义 + Git 历史，专为编程优化                      │
│                                                                  │
│  vs Vibe Kanban (项目管理)                                       │
│  ─────────────────────────────────────────────                   │
│  Vibe Kanban: 管理现在（看板、执行、审查）                         │
│  MemNexus: 连接过去与未来（记忆、追溯、复用）                      │
│                                                                  │
│  【关系】互补而非竞争：VK 负责执行，MemNexus 负责记忆              │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 四、目标用户

### 4.1 一级目标：独立开发者 / 小团队

**画像**:
- 使用 2+ 个 AI 编程工具（如 Kimi + Claude）
- 维护多个项目，经常切换上下文
- 希望 AI 能"记住"项目背景，减少重复解释

**痛点**:
- "每次开新 Session 都要重新解释项目结构"
- "上周写的功能，这周忘了为什么这样设计"
- "用不同 AI 工具，每个都要重新教一遍"

**价值主张**:
> "让你的 AI 编程助手记住项目的一切，无论你在用 Kimi、Claude 还是 Codex"

### 4.2 二级目标：Vibe Kanban 用户

**画像**:
- 已经在用 Vibe Kanban 管理 AI 编程任务
- 感受到 Session 之间缺乏记忆共享

**痛点**:
- "VK 的 Session 切换后，新 Agent 不知道之前做了什么"
- "同一个 Issue 的多个 Workspace 之间没有关联"

**价值主张**:
> "为 Vibe Kanban 添加长期记忆能力，让每个 Workspace 都能记住之前的决策"

### 4.3 三级目标：中文开发者生态

**画像**:
- 主要使用 Kimi CLI 等中文友好的 AI 工具
- 需要本土化的记忆管理解决方案

**机会**:
- 所有主流记忆层产品都是英文优先
- Kimi CLI 没有官方记忆基础设施

## 五、产品愿景

### 5.1 短期愿景（6 个月）

成为 **Vibe Kanban 用户首选的记忆插件**，解决 Session 记忆共享问题。

**成功标准**:
- 100+ 活跃用户
- 被 1 个知名开发者/团队推荐
- 用户反馈："这个插件帮我省了重复解释的时间"

### 5.2 中期愿景（1 年）

成为 **AI 编程工具的事实标准记忆层**，服务多个主流工具。

**成功标准**:
- 支持 Vibe Kanban + Kimi CLI + VSCode 扩展
- 1000+ 活跃用户
- 开源社区贡献者 10+

### 5.3 长期愿景（3 年）

成为 **代码智能的基础设施**，让每个 AI 编程工具都能理解代码的过去与现在。

**愿景描述**:
> 未来的 AI 编程助手，不再是"无状态"的。它们能记住：
> - 你三年前为什么选这个架构
> - 这个函数被重构过几次，每次的原因
> - 项目中的隐含约定和设计决策
> 
> MemNexus 是这个未来的记忆基础设施。

## 六、核心价值主张

### 6.1 对开发者

```
Before MemNexus:
你 → "帮我优化登录功能"
AI → "请问登录功能在哪里？用什么技术栈？有什么特殊要求？"
你 → （重复解释第 10 遍）

After MemNexus:
你 → "帮我优化登录功能"
AI → "根据记忆，登录功能在 auth/login.py，使用 JWT + Redis，
      上次修改是 2 周前（commit: a1b2c3），当时是为了解决并发问题。
      建议优化方向：..."
```

### 6.2 对团队

```
Before MemNexus:
新成员 → "这个函数为什么这么写？"
你 → "呃... 我记得当时有个原因..."

After MemNexus:
新成员 → "这个函数为什么这么写？"
AI → "根据项目记忆，这个函数在 3 个月前的 PR #123 中修改，
      当时讨论记录：'为了避免循环依赖，我们决定...'"
```

## 八、关键原则

1. **Code-First**: 一切设计围绕代码场景优化
2. **Git-Native**: Git 不是附件，是核心
3. **Tool-Agnostic**: 不绑定任何 AI 工具或平台
4. **Developer-First**: 开发者体验优先于技术先进性
5. **Open-Core**: 核心开源，商业化不损害开源用户
